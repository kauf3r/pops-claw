**Step 5: If Slack DMs did NOT arrive — implement fallback**

If the isolated session cannot send Slack DMs (known risk from research — MEDIUM confidence), activate the fallback: add a new `yolo-notify` cron job.

Fallback cron job spec:
- **Name:** `yolo-build-notify`
- **Schedule:** `45 8 * * * UTC` (= 12:45 AM PT — fires ~75 min after build starts, build should be done)
- **Session:** `main` (needs Slack access)
- **Kind:** `agentTurn`
- **Payload:** "Check /workspace/yolo-dev/yolo.db for the most recent build completed in the last 2 hours. SQL: SELECT name, status, self_score, description FROM builds WHERE started_at >= datetime('now', '-2 hours') AND status IN ('success','partial','failed') ORDER BY id DESC LIMIT 1. If a build is found, send a Slack DM to D0AARQR0Y4V: 'YOLO build complete: {name} — {status}, score {self_score}/5 | {description}'. If no build found, send nothing."

Register fallback cron:
```bash
/home/ubuntu/.npm-global/bin/openclaw cron add \
  --name "yolo-build-notify" \
  --expr "45 8 * * *" \
  --session main \
  --message "Check /workspace/yolo-dev/yolo.db for the most recent build completed in the last 2 hours. SQL: SELECT name, status, self_score, description FROM builds WHERE started_at >= datetime('now', '-2 hours') AND status IN ('success','partial','failed') ORDER BY id DESC LIMIT 1. If a build is found, send a Slack DM to D0AARQR0Y4V: 'YOLO build complete: {name} -- {status}, score {self_score}/5 | {description}'. If no build found, do nothing."
```

Note: The fallback fires once daily regardless of whether a build ran — the SQL handles the "no build" case cleanly.
