**Step 2 -- Prepare sandbox-compatible config.json:**

CRITICAL (Pitfall 2): The host ~/.gbrain/config.json has host paths. Inside sandbox, paths are different. The config directory itself gets bind-mounted to `/home/node/.gbrain/`, and `~/clawd/db/` is already mounted at `/workspace/db/`. So the database_path inside config.json must use the SANDBOX path.

Create or update ~/.gbrain/config.json so database_path uses the sandbox-compatible path:
```json
{
  "engine": "pglite",
  "database_path": "/workspace/db/gbrain/brain.pglite"
}
```

NOTE: This means when running gbrain on the HOST (outside sandbox), the config will point to a sandbox path that doesn't exist on the host. After this change, host-side gbrain invocations will need `--path` override or a separate config approach. Since the primary use case is sandbox (Bob), this is acceptable. If host-side use is needed later, use:
```bash
gbrain --path ~/clawd/db/gbrain/brain.pglite <command>
```
Or create a host wrapper. For now, optimize for sandbox use case.

**Step 3 -- Determine bind-mount strategy based on Plan 01 result:**

**IF Path A succeeded (compiled binary works):**
Add to `agents.defaults.sandbox.docker.binds`:
```
"/home/ubuntu/gbrain/bin/gbrain-linux-x64:/usr/local/bin/gbrain:ro"
"/home/ubuntu/.gbrain:/home/node/.gbrain:rw"
```
The `~/clawd/db/` -> `/workspace/db/` mount already exists, so PGLite data at `/workspace/db/gbrain/brain.pglite` is already accessible. No additional data mount needed.

**IF Path B needed (Bun runtime, compiled binary failed):**
Create wrapper script:
```bash
cat > ~/scripts/gbrain-wrapper.sh << 'WRAPPER'
#!/bin/sh
exec /usr/local/bin/bun /opt/gbrain/src/cli.ts "$@"
WRAPPER
chmod +x ~/scripts/gbrain-wrapper.sh
```

Add to `agents.defaults.sandbox.docker.binds`:
```
"/home/ubuntu/.bun/bin/bun:/usr/local/bin/bun:ro"
"/home/ubuntu/gbrain:/opt/gbrain:ro"
"/home/ubuntu/scripts/gbrain-wrapper.sh:/usr/local/bin/gbrain:ro"
"/home/ubuntu/.gbrain:/home/node/.gbrain:rw"
```
